# Definition
Innovation Studio is a production from BMC Helix.  
BMC Innovation Studio leverages BPM (Business Process Model) to allow users to design and automate their business processes, in a Process Definition.    
A process is a set of actions that are executed in a specific order, it can be used to automate a business process, for example to automate the onboarding of a new employee, or to automate the incident management process.    
A process can have input parameters that can be provided when triggering the process, and output parameters that can be retrieved after the process execution is completed.
  
The goal of this instruction is to explain how to create a new Innovation Studio process by using a template json file, and by filling the variable list explained below.


# Pre-Creation Checklist

Before creating a process definition, verify the following to prevent common errors:

- [ ] **Process name is unique**
  - Check: List all `.def` files in `/package/src/main/definitions/db/process/`
  - Ensure your intended process name doesn't already exist
  - Command (macOS/Linux): `find /package/src/main/definitions/db/process/ -name "*.def" -exec grep -H "name.*:" {} \;`
  - Command (Windows PowerShell): `Get-ChildItem "package\src\main\definitions\db\process\" -Filter "*.def" | Select-String "name"`

- [ ] **All GUID will be unique**
  - Generate a new UUID/GUID when asked (don't reuse existing ones)
  - Command to check (macOS/Linux): `find /package/src/main/definitions/db/process/ -name "*.def" -exec grep -H "<your-guid>" {} \;`
  - Command to check (Windows PowerShell): `Get-ChildItem "package\src\main\definitions\db\process\" -Filter "*.def" | Select-String "<your-guid>"`
  - Should return no results

- [ ] **Process activity exists and is correct**
  - Verify the Process Activity name you plan to use in the process definition has been generated by you and matches the value in the `@Action()` decorator `name` property of the class file you created,

- [ ] **Input parameters match component definition**
  - Verify parameter names in the process activity signature match,
  - Check the `@ActionParameter()` decorators `name` properties of the process activity class you generated
  - Ensure spelling and casing match exactly

- [ ] **Timestamp is current**
  - Use `Date.now()` or equivalent to get current milliseconds since epoch
  - Should be a reasonable value close to current time

- [ ] **File naming is correct**
  - Format: `<ProcessName>.def` (use the process name without bundle ID prefix)
  - Location: `/package/src/main/definitions/db/process/`
  - Example: For process `com.example.sample-application:Order Pizza`, filename is `Order Pizza.def`


# Template file description
The templates file is a json files that contains the definition of an Innovation Studio process containing the process activity mentioned or created, with some variables to fill in.  
The variables are explained in the variable list section below.  
Here is the template file:
* `process.json`: can be used as a starting point to create a new Innovation Studio process, by filling in the variables with the appropriate values, and by modifying the process definition as needed, 
  * The template file can be found in the same folder as this instruction file, with the name `process.json`.

## Process properties (`process.json`)
Here are the important properties of the Process definition:
* `name`: The fully qualified name of the process, in the format `<bundleId>:<process name>`,
* `guid`: The guid of the process, it should be a unique identifier,
* `flowElements`: The list of process activities that are used in the process,
  * The flowElements described later on are specifically designed to only support one process activity, with the following flow:
    * Start Event -> Sequence Flow 1 -> Process Activity -> Sequence Flow 2 -> End Event,
* `layout`: The layout of the process, it defines the structure of the process and the process activities that are used in the process,
* `inputParams`: The list of input parameters that are used in the process, these input parameters can be used in the process activities, and they can be provided when triggering the process in the code you will generate.
* `outputParams`: The list of output parameters that are used in the process, these output parameters can be set in the process activities, and they can be retrieved after the process execution is completed in the code you will generate.

## Building a full process definition
To build a full process definition, you can start from the `process.json` template file, and fill in the variables with the appropriate values.  


# Variable list to create a new Innovation Studio process from the template (`process.json`)
You can read a template of an Innovation Studio process instruction file here: [Template File](./process.json),

### Variable list
#### Process definition properties variables
* <timestamp>: when the process is created, it should be the current timestamp as milliseconds since Unix epoch (January 1, 1970, 00:00:00 UTC),
  * **Format**: A number like `1771460504496` (not an ISO date string)
  * **How to generate**: 
    * JavaScript/TypeScript: `Date.now()`
    * Command line (macOS/Linux): `date +%s%3N`
    * Command line (Windows PowerShell): `[DateTimeOffset]::UtcNow.ToUnixTimeMilliseconds()`
  * **Example**: `1771460504496` represents 2026-02-19T00:21:44.496Z
* <processFullyQualifiedName>: The fully qualified name of the process that you will create, in the format `<bundleId>:<process name>`,
  * **Critical** :
    * The process name should be unique across all processes in the system, otherwise the processiew creation will fail. You can check if the process name already exists by listing all processes in the folder `/package/src/main/definitions/db/process/`,
    * You can check the instruction file [GettingProcessDefinition.md](./GettingProcessDefinition.md) for more information.
* <processGuid>: The guid of the process, it should be a unique identifier, it should have this format `008904ea-fb8e-41a1-84ce-1b64f8e7a754`,
  * **How to generate**: You can use any UUID v4 generator:
    * TypeScript: `crypto.randomUUID()`
    * Command line (macOS/Linux): `uuidgen | tr '[:upper:]' '[:lower:]'`
    * Command line (Windows PowerShell): `[guid]::NewGuid().ToString().ToLower()`
    * Online: https://www.uuidgenerator.net/
  * **Critical** :
    * The process guid should be unique across all processes in the system, otherwise the process creation will fail. You can check if a process guid already exists by checking process guids listing all processes in the folder `/package/src/main/definitions/db/process/`,
    * You can check the instruction file [GettingProcessDefinition.md](./GettingProcessDefinition.md) for more information.
  * **Critical** :
    * Be very careful to only replace `<processGuid>` by the generated guid, as you can see in the template, the real guid of the process has a leading `rx-` that should not be removed,


#### Out Of The Box Process Activities variables
Some process activities are provided out of the box in the template to generate this flow:
Start Event -> Sequence Flow 1 -> Process Activity -> Sequence Flow 2 -> End Event
We need to generate guids for each of them. A guid should be a unique identifier, it should have this format `008904ea-fb8e-41a1-84ce-1b64f8e7a754`,
* **How to generate**: You can use any UUID v4 generator:
  * TypeScript: `crypto.randomUUID()`
  * Command line (macOS/Linux): `uuidgen | tr '[:upper:]' '[:lower:]'`
  * Command line (Windows PowerShell): `[guid]::NewGuid().ToString().ToLower()`
  * Online: https://www.uuidgenerator.net/
  
Here are the variables to fill in for these out of the box process activities:
* Start Event
  * <startEventGuid>: guid of the Start Event process activity, generate a new one,
* Sequence Flow 1
  * <sequenceFlow1Guid>: guid of the first Sequence Flow process activity, generate a new one,
* Process Activity
  * This is the process activity that you created, we will adress its properties in the next chapter,
* Sequence Flow 2
  * <sequenceFlow2Guid>: guid of the second Sequence Flow process activity, generate a new one,
* End Event
  * <endEventGuid>: guid of the End Event process activity, generate a new one,

**Critical** :  
Be very careful to only replace the different variables by the generated guid, as you can see in the template, the real guid of those objects has a leading `rx-` that should not be removed.


#### Created process activity variables
Here are the variables to fill in for the process activity that you created and that you want to use in the process definition:
* <processActivityName>: the name of the process activity — copy the exact string value from the `name` property of the `@Action()` decorator of the process activity class. For example, for the annotation below the name is `orderPizza` (not `OrderPizza`):
  * ```java
    @Action(name = "orderPizza")
    public void orderPizza(
        @ActionParameter(name = "pizzaName") String pizzaName,
        @ActionParameter(name = "price") String price
    )
    ```
* <processActivityFullyQualifiedName>: The fully qualified process activity name, it should match the value of the `name` property in the `@Action()` decorator of the process activity class you created, prefixed by the bundle id and a ":", for example if your bundle id is `com.example.sample-application` and if you have a process activity class with the following method `orderPizza` decorated by the `@Action()` decorator, the fully qualified name should be `com.example.sample-application:orderPizza`:
  * ```java
    @Action(name = "orderPizza")
    public void orderPizza(
        @ActionParameter(name = "pizzaName") String pizzaName,
        @ActionParameter(name = "price") String price
    )
    ```
* <codedProcessActivityGuid>: the guid of the process activity, it should be a unique identifier, it should have this format `008904ea-fb8e-41a1-84ce-1b64f8e7a754`, you can generate it in the same way as explained above for the other guids, and be careful to only replace the variable by the generated guid, as you can see in the template, the real guid of the process activity has a leading `rx-` that should not be removed,
  * **How to generate**: You can use any UUID v4 generator:
    * TypeScript: `crypto.randomUUID()`
    * Command line (macOS/Linux): `uuidgen | tr '[:upper:]' '[:lower:]'`
    * Command line (Windows PowerShell): `[guid]::NewGuid().ToString().ToLower()`
    * Online: https://www.uuidgenerator.net/
* <processActivityInputMap>: the input map of the process activity,
  * It should match the input parameters of the process activity method that you created, decorated by the `@ActionParameter` decorator.
  * **Always use process input variable references** (defined in `inputParams`) as the expression values — do not hardcode literal values. This lets callers pass runtime values into the process.
    * For example if your process activity method has the following signature, you should generate two input map entries for `pizzaName` and `price`:
      * ```java
        @Action(name = "orderPizza")
        public void orderPizza(
            @ActionParameter(name = "pizzaName") String pizzaName,
            @ActionParameter(name = "price") String price
        )
        ```
      * The format of an input map entry is:
        * ```json
          {
            "assignTarget" : "<actionParameterName>",
            "expression" : "${processContext.<processInputVariableName>}"
          }
          ```
        * Where `<actionParameterName>` is the exact `name` value from `@ActionParameter`, and `<processInputVariableName>` is the `name` of the corresponding process input variable you created in `inputParams`.
        * **⚠️ CRITICAL: Use the process input variable NAME, not the ID!**
          * ✅ **CORRECT**: `${processContext.Pizza Name}` or `${processContext.Price}` (using the `name` field)
          * ❌ **WRONG**: `${processContext.536870913}` (using the `id` field)
          * The `id` field (e.g., 536870913) is only used internally in the parameter definition itself — never in expressions!
      * Example for the method above (assuming process input variables named `Pizza Name` and `Price`):
        * ```json
          "inputMap" : [ {
            "assignTarget" : "pizzaName",
            "expression" : "${processContext.Pizza Name}"
          }, {
            "assignTarget" : "price",
            "expression" : "${processContext.Price}"
          } ]
          ```

#### Input Parameter Value Formatting (alternative: hardcoded literals)
> **Note**: Using process input variable references (e.g. `${processContext.Pizza Name}`) is always preferred. Only use hardcoded literals when you intentionally want a fixed value regardless of caller input.

If you do need to hardcode a literal value in an `expression`, the value must be a **JSON-escaped string**:

| Value Type     | Example Value                        | Format in expression             | Explanation                                                          |
|----------------|--------------------------------------|----------------------------------|----------------------------------------------------------------------|
| String literal | `Hello World`                        | `"\"Hello World\""`              | Outer quotes = JSON string, inner escaped quotes = string literal    |
| Number         | `42`                                 | `"42"`                           | Number as string (no inner quotes)                                   |
| Boolean        | `true`                               | `"true"`                         | Boolean as string (no inner quotes)                                  |
| Null           | `null`                               | `null`                           | null                                                                 |
| Array          | `["desktop", "tablet"]`              | `"[\"desktop\",\"tablet\"]"`     | Escaped JSON array                                                   |
| Object         | `{"key": "value"}`                   | `"{\"key\":\"value\"}"`          | Escaped JSON object                                                  |

Example with hardcoded literals:
```json
"inputMap" : [ {
  "assignTarget" : "pizzaName",
  "expression" : "\"Margherita\""
}, {
  "assignTarget" : "price",
  "expression" : "\"15.99\""
} ]
```


# Generating process variables
## Definition
A process definition can have input and output variables.  
* The process input variables allows other process or an UI element to pass values at runtime that can be used in the process,
  * They are defined in the process definition, in the `inputParams` property, which is an array of objects,
  * There can be several process input parameters,
* The process output variables allows the process to return values to the caller process or UI element,
  * They are defined in the process definition, in the `outputParams` property, which is an array of objects,
  * There can be several process output parameters,

## How to build a process variable?
* The important properties are:
  * `name`: the name of the process input variable,
    * **Critical** this name needs to be unique across all input and output variables of the current process definition, otherwise the process creation will fail. You should only use alphanumeric characters and spaces in the name to avoid any issue, for example `Pizza Name` or `Price`,
  * `resourceType`: the type of the variable, for example `com.bmc.arsys.rx.standardlib.record.CharacterFieldDefinition`,
    * **Critical** check the following chapter `Process variable types` for the list of available types, only use those types, 
  * `description`: a description of the variable
  * `id`: the identifier of the process variable, it is an integer which starts at `536870913` and needs to be incremented at each new process variable, whether it is an input or output parameter,
    * **Critical** the id must be unique across the process variables,

## Generating input process variables
Generate as many process input variables as process activity inputs, matching types. For each process activity, read its template as per the `Template Example` column above and replace the necessary variables:
* `<timestamp>`: the timestamp you already generated for the process definition creation,
* `<processVariableName>`: the name of the process input or output variable,
  * **Critical** this name needs to be unique across all input and output variables of the current process definition, otherwise the process creation will fail. You should only use alphanumeric characters and spaces in the name to avoid any issue, for example `Pizza Name` or `Price`,
* `<processVariableDescription>`: a description of the process input or output variable,
* `<processVariableId>`: the identifier of the process variable, it is an integer which starts at `536870913` and needs to be incremented at each new process variable, whether it is an input or output parameter,
  * **Critical** the id must be unique across the process variables,
  
_Note:_  
If you cannot determine which type to use, use the Text type (`com.bmc.arsys.rx.standardlib.record.CharacterFieldDefinition`).  


Once you have the list of all input process variables:
* Add them to the process definition `inputParams` array,
* Leverage those process input variables in the process activity `inputMap` which is an array of objects, where here `<inputParameterValue>` will be a process input variable reference:
  * ```json
      {
        "assignTarget" : "<procesActivityInputParameterName>",
        "expression" : "<inputParameterValue>"
      }
      ```
* To reference a process input variable, use the syntax `${processContext.<input parameter name>}`,
  * **⚠️ IMPORTANT**: `<input parameter name>` refers to the `name` field of the process input variable (e.g., "Pizza Name"), **NOT** the `id` field (e.g., 536870913).
  * For example:
    * If you have a process activity with an input parameter `pizzaName`, you might have generated a process input variable with the name `Pizza Name` and the id `536870913`,
    * ✅ **CORRECT**: You can use it in the process activity input map with the value `${processContext.Pizza Name}` (using the name),
    * ❌ **WRONG**: Do NOT use `${processContext.536870913}` (using the id),
      * ```json
        "inputMap" : [ {
          "assignTarget" : "pizzaName",
          "expression" : "${processContext.Pizza Name}"
        }]
         ```

##  Generating output process variables

### Step 1: Determine what the process activity returns
Look at the return type of the `@Action`-annotated method in the Java file:

| Return type                       | What to do                                                                                              |
|-----------------------------------|---------------------------------------------------------------------------------------------------------|
| `void`                            | **Do not create any output variables.** Leave `outputParams` and `outputMap` as empty arrays `[ ]`.    |
| Primitive / `String`              | Create **one** output variable whose type matches the primitive (e.g. `IntegerFieldDefinition`).        |
| A **class** (e.g. `OrderResponse`)| **Read the class Java file.** Create **one output variable per `@JsonProperty` member** of that class. |
| A **`List<SomeClass>`**           | Treat it as a single Object output variable (`ObjectFieldDefinition`) representing the full list.       |

### Step 2: Read the return class (if applicable)
If the method returns a class:
1. Find the class file in the bundle Java sources (e.g. `bundle/src/main/java/.../<ClassName>.java`).
2. Identify every field annotated with `@JsonProperty` — those are the members that will be accessible in the process output.
3. For each such member, create one process output variable, choosing the correct type from the table in `Process variable types`.

**Example**: if the method returns `OrderResponse` and that class has:
```java
@JsonProperty private String orderId;
@JsonProperty private int itemCount;
@JsonProperty private List<OrderItem> items;
```
→ create three output variables: `Order Id` (CharacterFieldDefinition), `Item Count` (IntegerFieldDefinition), `Items` (ObjectFieldDefinition).

### Step 3: Build the output variable entries
For each output variable, read its template as per the `Template Example` column in the `Process variable types` table and fill in:
* `<timestamp>`: the timestamp you already generated for the process definition creation,
* `<processVariableName>`: a human-readable name, **unique** across all input AND output variables of this process,
  * Use only alphanumeric characters and spaces, e.g. `Order Id`, `Item Count`,
* `<processVariableDescription>`: a short description of the variable,
* `<processVariableId>`: integer starting at `536870913`, incremented for **every** variable (input or output),
  * **Critical**: must be unique across all process variables in this definition.

_Note:_ If you cannot determine the correct type, default to Text (`com.bmc.arsys.rx.standardlib.record.CharacterFieldDefinition`).

### Step 4: Wire up the outputMap
Once you have all output variables:
* Add them to the process definition `outputParams` array.
* For each output variable, add an entry to the `outputMap` of the `ServiceTaskDefinition` flow element:
  ```json
  {
    "assignTarget" : "<processOutputVariableName>",
    "expression" : "<outputParameterExpression>"
  }
  ```
* **⚠️ CRITICAL: Use the process output variable NAME for assignTarget, not the ID!**
  * `<processOutputVariableName>` must be the `name` field of the output variable (e.g., "Total Count", "Full Price")
  * ✅ **CORRECT**: `"assignTarget" : "Total Count"` (using the `name` field)
  * ❌ **WRONG**: `"assignTarget" : "536870915"` (using the `id` field)
  * The `id` field is only used in the output variable definition itself — never as an assignTarget!

**How to build `<outputParameterExpression>`:**

| Scenario | Expression format | Example |
|---|---|---|
| Return type is a **class**, accessing a scalar member | `${activityResults.<fullGuid>.output.<memberName>}` | `${activityResults.rx-abc123.output.orderId}` |
| Return type is a **class**, member is itself a `List<...>` | `${activityResults.<fullGuid>.output.<memberName>}` | `${activityResults.rx-abc123.output.items}` |
| Return type is a **`List<SomeClass>`** directly | `${activityResults.<fullGuid>.output[0].<memberName>}` to access the first element | `${activityResults.rx-abc123.output[0].orderId}` |

Where:
* `<fullGuid>` is the **full** guid of the `ServiceTaskDefinition` flow element, **including the `rx-` prefix** (e.g. `rx-902b4266-3bba-4732-890a-e25bbd2cb2d0`).
* `<memberName>` is the exact Java field name of the `@JsonProperty`-annotated member in the return class.

**Critical**:
* If the process activity returns `void`, leave both `outputParams` and `outputMap` as `[ ]`.
* The `rx-` prefix on the guid in `activityResults` expressions is **mandatory** — do not omit it.

**Full example** — `addFruitRecord` returns `FruitListResponse` with members `totalCount` (int) and `fruits` (List):
```json
"outputMap" : [ {
  "assignTarget" : "Total Count",
  "expression" : "${activityResults.rx-902b4266-3bba-4732-890a-e25bbd2cb2d0.output.totalCount}"
}, {
  "assignTarget" : "Fruits",
  "expression" : "${activityResults.rx-902b4266-3bba-4732-890a-e25bbd2cb2d0.output.fruits}"
} ]
```
And the corresponding `outputParams`:
```json
"outputParams" : [ {
  "resourceType" : "com.bmc.arsys.rx.standardlib.record.IntegerFieldDefinition",
  ...
  "name" : "Total Count",
  "id" : 536870915,
  ...
}, {
  "resourceType" : "com.bmc.arsys.rx.services.process.domain.record.ObjectFieldDefinition",
  ...
  "name" : "Fruits",
  "id" : 536870916,
  ...
} ]
```


# Common Mistakes to Avoid

When creating process definitions, pay special attention to these frequent errors:

## ❌ Mistake #1: Using Process Variable IDs Instead of Names

**The Problem:**
- Process variables have both a `name` field (e.g., "Price", "Total Count") and an `id` field (e.g., 536870913, 536870915)
- The `id` field is **only** used in the variable definition itself
- The `name` field is used in **all expressions and references**

**Wrong Examples:**
```json
// ❌ WRONG - Using ID in inputMap expression
"inputMap" : [ {
  "assignTarget" : "price",
  "expression" : "${processContext.536870913}"  // ❌ Wrong - this is the ID
} ]

// ❌ WRONG - Using ID in outputMap assignTarget
"outputMap" : [ {
  "assignTarget" : "536870915",  // ❌ Wrong - this is the ID
  "expression" : "${activityResults.rx-abc123.output.fullPrice}"
} ]
```

**Correct Examples:**
```json
// ✅ CORRECT - Using NAME in inputMap expression
"inputMap" : [ {
  "assignTarget" : "price",
  "expression" : "${processContext.Price}"  // ✅ Correct - this is the name
} ]

// ✅ CORRECT - Using NAME in outputMap assignTarget
"outputMap" : [ {
  "assignTarget" : "Full Price",  // ✅ Correct - this is the name
  "expression" : "${activityResults.rx-abc123.output.fullPrice}"
} ]
```

**Quick Reference:**
| Location | Field to Use | Example |
|----------|-------------|---------|
| `inputParams` / `outputParams` definition | Both `name` AND `id` | `"name": "Price"`, `"id": 536870913` |
| `inputMap` expression | `name` only | `${processContext.Price}` |
| `outputMap` assignTarget | `name` only | `"assignTarget": "Full Price"` |

## ❌ Mistake #2: Forgetting the `rx-` Prefix in outputMap Expressions

**Wrong:**
```json
"expression" : "${activityResults.abc123.output.value}"  // ❌ Missing rx- prefix
```

**Correct:**
```json
"expression" : "${activityResults.rx-abc123.output.value}"  // ✅ Has rx- prefix
```
